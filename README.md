# 共享单车调度优化问题

## 题目说明

本项目围绕共享单车站点运行数据，完成以下 4 个问题：

### 问题1：描述性分析

基于 `附件2_每小时借还记录.csv` 与 `附件1_站点基础信息.csv`，分析 30 个站点在一周内的借还规律，识别：

1. 全天典型借出峰、归还峰及持续时间；
2. 借出量最高 / 最低站点及其空间分布特征；
3. 工作日与休息日借还模式的主要差异。

### 问题2：未来 1 天需求预测

建立对目标日期 24 小时逐站点借出量、归还量的预测模型，并在测试日上评估精度。

当前实现采用：

- 小时周期特征：`hour_sin`、`hour_cos`
- 日期类型特征：`is_weekday`
- 可提前获得的历史特征：前一日同小时借还量 `lag_24`
- 训练集均值基线：同站点同小时历史均值
- 空间特征：站点聚类分区 `zone`

最终预测值采用“基线 + 残差模型”的方式得到。

### 问题3：调度优化

基于问题2的预测结果，对工作日进行“无调度 / 有调度”仿真对比，输出：

1. 总费用；
2. 满桩 / 空桩发生次数；
3. 用户无法借车 / 还车次数；
4. 用户需求满足率；
5. 调度明细与库存变化。

当前题3实现采用离散时延调度：

- 调度在整点发车；
- 调度时间按 `t_ij = round(d_ij / v)` 取整；
- 源站发车当小时立即扣减库存；
- 目标站在 `t + t_ij` 时刻入库。

### 问题4：模型推广与灵敏度分析

在问题3的基础上，完成两部分扩展：

1. 自行新增 5 个站点，构造扩展后的 35 站网络，并重新运行调度模型；
2. 对运输成本、搬运成本、满桩惩罚系数、空桩惩罚系数做敏感性分析。

## 本次检查后确认的关键问题

### 已修正的问题

1. 问题2原实现存在明显数据泄漏。  
   原脚本先对全量数据做基线与滞后特征，再拆训练 / 测试，等于在预测目标日时使用了目标日本身的信息。现在改为：先按目标日期切分，再仅用训练集构造基线。

2. 问题2原实现错误使用了 `lag_1 / lag_2 / lag_3 / rolling`。  
   这些特征在“整天提前预测”场景下不可直接获得，会隐式使用目标日真实值。现在保留仅在预测开始前可知的 `lag_24` 及静态特征。

3. 问题2原 MAPE 指标失真严重。  
   原公式在真实值为 0 时会产生极大数值，现在改为仅在非零真实值上计算 MAPE。

4. 问题1输出文件全部堆在根目录。  
   现已统一输出到 `outputs/problem1/`，并去掉了工作日固定除以 5、休息日固定除以 2 的硬编码假设。

5. 问题3自动读取预测文件的逻辑不稳定。  
   原实现会直接读取根目录第一个匹配到的 `predictions_*.csv`。现在优先读取 `outputs/problem2/` 中排序后的最新预测文件。

6. 问题3运输成本公式不合理。  
   原实现把“公里成本 + 单车搬运成本”整体再乘车辆数，导致里程成本被重复放大。现在改为：  
   `运输成本 = 每公里成本 × 距离 + 每辆搬运成本 × 搬运数量`

7. 问题3原实现默认瞬时调度。  
   现已改为离散时延到站：调度在出发时刻扣减源站库存，在 `t + round(d_ij / v)` 时刻进入目标站库存，其中 `v` 为附件3给出的调度车速。

8. 根目录存在多份伪代码说明和重复总结，且文件扩展名混乱。  
   已统一归档至 `docs/archive/`。

### 仍需在论文 / 报告中说明的限制

1. 问题2样本量只有 7 天，模型精度上限天然受限，结论更适合做课程 / 建模题演示，不适合直接外推到真实运维。

2. 问题3当前采用启发式贪心调度，不是严格数学规划最优解；如果答辩里强调“最优”，需要明确这是“近似优化方案”。

3. 问题3的“满桩惩罚”与“用户还车失败”之间仍存在建模口径差异。当前程序已能稳定对比方案优劣，但若要完全贴合题面，建议进一步把惩罚定义写成严格的数学口径。

## 问题3修订说明

下面两点已根据题目原文完成修订，代码与文档口径现已统一。

### 1. 空桩惩罚成本的定义

题目原文给的是：

- 空桩惩罚成本 = 惩罚系数 × 低于安全库存的车辆数

因此严格按题面，空桩惩罚应写成：

```text
空桩惩罚 = beta × max(0, 安全库存 - 当前库存)
```

而不是：

```text
空桩惩罚 = beta × max(0, 借车需求 - 当前库存)
```

核查结论：

- 当前代码本来就是按题面实现的，使用的是“安全库存缺口”。
- 现已将 README / 主程序说明 / Word 文档中的相关表述统一修正为这一口径。

当前代码口径：

- 见 `problem3_simulation.py` 中 `calculate_penalty()`：
  `under = max(0, safe_inventory - S_end)`

本次已完成的修改：

1. 保持代码中的空桩惩罚实现不变。
2. 将所有说明文字统一为：
   - 借车失败量：`max(0, 借车需求 - 当前库存)`
   - 空桩惩罚：`max(0, 安全库存 - 当前库存)`
3. 在问题3主程序说明中补充了“空桩惩罚口径”说明，避免再次混淆。

### 2. 货车容量约束 `C_max = 15` 的解释

附件3原文是：

- 货车最大载车量：`15 辆/次`

这句更自然的理解是：

- 每一次调度 / 每一趟运输，最多搬运 15 辆；
- 即单条调度记录应满足 `x_ijt <= 15`。

但当前代码采用的是更强的隐藏假设：

- 在每个整点时刻，对全网络所有调度加总后，总搬运量最多为 15 辆；
- 也就是把 `15 辆/次` 实现成了“每小时全局总调度量不超过 15”。

当前代码位置：

- `problem3_simulation.py` 的 `greedy_schedule_enhanced()` 中，
  `remaining_capacity` 初始设为 `truck_capacity = 15`，
  然后同一小时内所有调度共享这一上限。

这意味着程序实际上默认了：

- 每个整点只有一辆车；
- 且该车在该整点的所有可执行调度总共只能搬 15 辆。

这个假设题目并没有明确给出，因此存在口径风险。

修订判断：

- 题面与附件3更自然的解释应为“单次运输最多 15 辆”，而不是“每小时全局总调度量最多 15 辆”。
- 因此原代码约束偏严，现已改为“单条调度记录不超过 15 辆”。

本次已完成的修改：

1. 数学模型口径改成单次调度约束：

```text
0 <= x_ijt <= 15
```

2. 在 `problem3_simulation.py` 中取消“每小时全局合计最多 15 辆”的共享容量限制。
3. 每小时仍允许多条调度记录，但每条记录都满足 `x_ijt <= 15`。
4. README、主程序说明和 Word 文档均同步改成这一口径。
